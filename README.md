# AI Office Agent

## 项目简介

AI Office Agent 是一个面向办公场景的智能体后端原型项目，目标是构建一个能够进行多轮对话、按需调用工具、检索本地知识库并结合实时网页信息回答问题的系统。

当前版本聚焦于打通“聊天 + 工具调用 + RAG + API 服务”的主链路，使系统具备一个可运行的最小可用版本（MVP），并为后续扩展到邮件发送、任务规划、日历集成、前端界面以及 Windows 桌面端部署打下基础。

与单纯的聊天机器人或只做文档问答的项目不同，这个项目更强调**面向真实应用的系统化设计**：将模型接入、工具注册、知识检索、会话管理和 API 服务拆分为相互独立的模块，便于后续替换、扩展和工程化演进。

---

## 为什么做这个项目


在很多快速搭建的 Agent 项目中，模型调用、工具执行、检索逻辑和接口层常常被直接耦合在一起，虽然短期内可以跑通功能，但随着需求增加，很快就会出现以下问题：

- 业务逻辑散落在各处，难以维护
- 更换模型、向量库或工具实现时代价很高
- 新增前端、桌面端或外部系统接入时需要大量重构
- 会话状态、工具能力和检索链路难以独立演进

因此，这个项目更关注的是：

- 如何把 Agent 系统拆分为清晰的层次结构
- 如何让工具调用与业务逻辑解耦
- 如何让 RAG 成为可独立扩展的能力模块
- 如何让系统具备后续走向产品化的可能性

换句话说，这个项目的重点不只是“调用了一个大模型”，而是尝试构建一个**具备可维护性、可替换性与可扩展性**的 Agent 后端骨架。

---

## 核心能力

### 1. 多轮会话管理
基于 `session_id` 保存对话上下文，支持连续对话，并为后续扩展持久化记忆、用户状态管理和长周期任务处理预留接口。

### 2. 工具调用机制
将工具抽象为统一接口，由 Agent 根据用户问题决定是否调用工具，而不是把业务逻辑硬编码进 Prompt 中。当前已支持邮件草拟、本地文档检索和网页搜索等能力。

### 3. RAG 检索链路
支持本地文档导入、文本切块、向量检索、重排序以及引用返回，能够让回答建立在用户提供的资料之上，而不是只依赖模型参数记忆。

### 4. 模型路由
根据任务复杂度和场景特征选择不同模型，在响应速度、成本和回答质量之间进行平衡，为后续多模型协同奠定基础。

### 5. API 化服务
通过 FastAPI 暴露标准接口，使系统既可以作为后端服务被网页前端调用，也便于后续封装为 Windows 客户端或接入其他业务系统。

### 6. 面向扩展的模块化设计
会话管理、Agent 编排、工具注册、文档检索、模型接入相互独立，方便未来替换底层实现，而不需要推翻整个系统重写。

## 架构设计

项目采用分层结构，而不是将框架调用直接散落在路由函数或单一脚本中。整体逻辑如下：

```text
Client / Frontend
        │
        ▼
   FastAPI API Layer
        │
        ▼
Service Orchestration Layer
├── Chat Service
├── RAG Service
└── Tool Service
        │
        ▼
   Agent Runtime Layer
   ├── Planning
   ├── Tool Execution
   └── Response Synthesis
        │
        ▼
Infrastructure Layer
├── LLM Provider / Router
├── Vector Store
├── Embedding Model
└── Tool Registry
```

### 分层说明

#### API Layer
负责对外暴露服务接口，例如聊天、知识库导入、文档问答、工具查看与运行等，是系统与前端或外部调用方的统一入口。

#### Service Layer
负责业务编排。它不直接承担底层实现细节，而是负责组织会话、调用 Agent、触发检索、管理工具执行结果，从而保证系统逻辑清晰可维护。

#### Agent Runtime Layer
负责“规划 → 工具调用 → 回答生成”的核心执行闭环。该层体现的是智能体的行为逻辑，而不是简单的一次性问答。

#### Infrastructure Layer
封装模型接入、Embedding、向量数据库、工具注册等底层能力，为上层提供可替换的基础设施支持。

### 这种设计的价值

这样的结构可以避免将业务逻辑直接绑定在具体框架或单一实现上，使系统在后续扩展时具备更好的灵活性，例如：

- 更换模型服务提供商而不影响上层业务
- 更换向量数据库而不重写 RAG 主流程
- 新增工具而不破坏原有接口
- 接入 Web 前端或桌面端而不重构核心后端逻辑

## 技术选型说明

当前版本在 Agent 编排部分使用了 **LangGraph**，在模型接入、Embedding、检索链路等部分复用了 **LangChain 生态中的部分组件**，以减少底层重复开发成本、尽快验证系统主链路。

但这个项目的核心价值并不在于“使用了某个 Agent 框架”，而在于：

- 系统边界由项目显式定义
- 服务分层由项目自行设计
- 工具抽象与注册方式由项目统一管理
- RAG 流程和 API 组织不是直接照搬框架默认模板

也就是说，在这个项目里，LangChain / LangGraph 更偏向于**基础设施与实现工具**，而不是项目本身的全部内容。

### 当前技术选择的原因

- **FastAPI**：适合作为后端接口层，便于调试、部署和后续接入前端
- **LangGraph**：适合表达 Agent 的状态流转和多步执行闭环
- **LangChain 组件**：在文档加载、Embedding、检索和模型调用兼容层方面提供了开发效率
- **Chroma**：适合作为本地原型阶段的向量数据库
- **OpenAI-compatible 接口**：便于兼容不同模型服务商，降低模型切换成本

## Current Status

当前版本是一个**可运行的 MVP（最小可用原型）**，已经打通了以下主链路：

- 聊天接口可用
- 支持基于 `session_id` 的多轮上下文管理
- 支持基础工具调用流程
- 支持本地文档导入与检索问答
- 支持网页搜索补充时间敏感信息
- 支持通过 API 方式对外提供服务


不过，当前版本仍然明确属于原型阶段，主要限制包括：

- 会话状态目前保存在进程内存中，尚未持久化
- 邮件能力目前仅支持草拟，未接入真实邮件服务
- RAG 仍以单机 Chroma 为主，索引管理较基础
- Web Search 为轻量实现，尚未加入更强的结果过滤与引用控制
- 工具数量仍较少，办公场景覆盖尚不完整
- 整体监控、评测、权限控制和异常追踪能力还不完善

因此，当前版本更适合作为一个**具备清晰架构和扩展价值的 Agent 后端原型**，而不是一个已经完全产品化的系统。

## Roadmap

接下来会围绕“更完整的办公助手能力”和“更强的工程化质量”两个方向持续优化：

- [ ] 将会话状态从内存升级为 Redis / PostgreSQL 持久化存储
- [ ] 接入真实邮件服务（Gmail / Outlook），支持发送、回复与线程管理
- [ ] 增加更多办公工具：日历、待办、会议纪要、文件总结、任务规划等
- [ ] 支持混合检索、查询改写、更稳定的 rerank 策略与更丰富的 metadata 管理
- [ ] 支持多知识库隔离与多用户场景
- [ ] 增加前端页面，并进一步封装为 Windows 桌面应用
- [ ] 将部分 Agent 运行链路逐步替换为更可控的自定义实现
- [ ] 引入日志、监控、评测和错误追踪机制，提升系统可观测性
- [ ] 补充测试用例、部署脚本和工程文档，提升项目可复现性
